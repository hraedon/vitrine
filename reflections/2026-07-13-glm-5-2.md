## On the project

vitrine's presentation layer is now genuinely maintainable. The 1732-line `render.py` monolith — which held templates-as-strings, projection logic, SVG assembly, and orchestration in one file — is gone, replaced by a clean DAG: `context.py` (the leaf of frozen typed view models) ← `projections/*.py` (one `project_*` per surface, the only place fact-ids resolve and ratios compute) ← `build.py` (orchestration only, 191 lines) → templates (formatting and branching only, reading a single `page` object). The architecture test still guards the import boundary (core never depends on site). The plan's "correctness by construction" philosophy — `mypy --strict`, `assert_never`, frozen dataclasses, the ancestry gate — pays dividends here: the byte-identical refactor was verifiable mechanically, and the one real bug (the `pushState` `:target` defect) was caught by the new browser tests, not by reasoning.

The plan's discipline around "the recovery must not reproduce the shape that caused the incident" (serialized WP landing on a single shared branch, one owner per file boundary, the ancestry gate as a hard invariant) is the reason this session could land six WPs cleanly. The `019-recovery-log.md` as a living document — each WP appending its landing note, decision points, and the byte-identity proof — is a pattern worth keeping for any future large refactor.

## On the work done

**What went well:**
- Subagents carried the mechanical bulk. WP2 (template extraction) and WP3+WP4 (the big split) were well-scoped tasks with a crisp byte-identity gate; both came back green on the first pass. The key was giving each subagent the *exact* target module layout from the plan, the line ranges to move, and the `diff -r` verification command as a hard gate. When the contract is "output must not change," subagents perform reliably because the feedback loop is instantaneous.
- The browser suite earned its keep immediately. WP5's subagent wrote the tests against the current DOM, found that 5 of them failed because `dismissOverlay()` used `history.pushState("#dismissed")` — which does not recalculate CSS `:target` in Chromium, leaving the overlay visually `display:flex` after Escape/close/backdrop. This was a real latent defect inherited from `9953a0e`, not a regression. The fix (`closeOverlay(); location.hash = "#dismissed"`) was one line, but finding it required actually running a headless browser. Without the suite, this would have shipped.
- The adversarial review (run via a `general` subagent since the dedicated reviewers were out of usage, per the user's note) found four minors and no criticals. The most useful catch was the `.gitignore`/ruff `build/` gap: setuptools' ephemeral `build/` dir mirrors `src/` on first install and surfaced 21 spurious E501s that the per-file-ignores didn't cover. That would have flaked the first CI run after any code change. Fixed with a two-line exclude.

**What was awkward:**
- The plan's §"DOM placement" wants overlay decks rendered outside `.wrap` in the static HTML. The current architecture renders them inline and `enhancements.js` moves the active overlay to `document.body` at runtime. This is functionally equivalent (browser-verified) but doesn't match the plan's preferred static structure. I deferred the static-DOM move rather than break byte-identity for marginal benefit — the macro split (`placard_card` + `placard_overlay` called separately) is now in place as the structural prerequisite, so a future session can do the move cleanly. This is a judgment call; a stricter reading of the plan would insist on it now.
- `FactRef` ended up in `context.py` rather than `projections/facts.py` (the task spec's first choice) to break a circular import (context → projections.facts → package `__init__` → projections.affordability → context). The subagent caught this and made the right call, but it's a reminder that the "leaf module" decision can't always be predicted from the plan's layout sketch.

**What I'm confident in:**
- Byte-identity is real — verified independently via `diff -r` at every WP boundary, via `git worktree` rebuild in the adversarial review, and via the wheel-install from outside the repo.
- The typed page boundary is structurally complete — the adversarial review confirmed all 8 page contexts + 14 view types are constructed by a `project_*` and consumed by a template (no dead types).
- The `:target` bug fix is correct — the hashchange it fires no-ops in `syncWithHash` (active is null), and the 5 previously-xfailed tests now pass green.

## Gaps to flag

- **The plan-019 branch is behind `main` on data.** Anyone merging PR #5 needs to decide: fast-forward `main` to the plan-019 tip (losing the post-`9953a0e` data additions on `main`), or rebase plan-019 onto `main` (re-porting the data commits onto the split architecture). The recovery log doesn't spell this out — it's a merge-strategy decision the user should make explicitly. The cleanest path is probably: merge plan-019 (bringing the architecture + the museum UI), then cherry-pick/re-port the data-only commits from `main`.
- **`enhancements.js` moves overlays to `document.body` at runtime via a comment-placeholder swap.** This works (browser-verified) but means the DOM the user sees with JS on differs from the DOM in view-source. A future refactor that places overlays outside `.wrap` statically would make the JS simpler and the DOM more legible. Not a bug — just a "this is more complex than it needs to be" note.
- **`tests/test_browser.py` uses `page.wait_for_timeout(...)` in several places** (50–150ms) to let the synchronous JS state machine settle. These could theoretically flake on a very slow CI runner, though 35/35 passed in ~17s locally. If CI flakes, bumping timeouts or using `expect(locator).to_be_visible()` (which auto-retries) more aggressively would harden them.
- **The `evaluate_room`/`affordability_for_room` double-compute** in `build.py` (once for the manifest, once inside `project_room`) is acceptable for 13 rooms but is a real redundancy. If the corpus grows large, hoisting the computation is a minor optimization.
- **`projections/metrics.py:load_recessions` reads `data/recessions.toml` directly** — the only projection module that touches the filesystem. Flagged in the recovery log; functionally correct, but it's the one place the ownership boundary is softer than the plan specifies.
