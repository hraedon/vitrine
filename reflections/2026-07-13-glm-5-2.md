## On the project

The fact model's "correctness by construction" philosophy continues to pay dividends. The `assert_never` dispatch over `DerivedOp` meant that adding `PRODUCT` and `QUANTITY_RATIO` immediately broke every site that needed to handle them — `_op_value`, the gate, the derived_placard macro — which is exactly the kind of mechanical safety the architecture test was designed to provide. The frozen-dataclass + closed-enum pattern makes extension safe in a way that open-ended dictionaries never could.

The cross-room resolution was the most consequential design decision. The fact model was built around room-local resolution ("operands must resolve to structured facts in the same room"), and WI-5 broke that assumption. The solution — a `fact_index` dict threaded through `evaluate` → `evaluate_room` → `build.py` — is backward-compatible (the parameter is optional, defaulting to None) and consistent with how `series` is already threaded. But it's a real architectural shift: derived facts can now reference facts in other rooms, which means the gate's `all_facts` lookup and the build's `fact_index` must stay in sync. The `all_facts` dict in `_check_derived` is rebuilt per-room (O(R^2) at current scale, not a concern), which is a minor wart but acceptable.

**What was awkward:**
- The 1950s weekly-earnings migration exposed a real tension in the fact model. The displayed hourly rate ($1.32 = 132 cents) is a rounded annual average, but the actual weekly earnings ($53.29) was computed from unrounded monthly values. The derived fact computes $53.46 (from the rounded inputs), which differs from both the old display value ($53.29) and the BLS series value ($53.33). I chose to accept $53.46 as the honest computation from the stored inputs, with notes explaining the discrepancy. This is the right call architecturally (the fact model stores what it stores), but it means the displayed value changed, which required updating cascading references in week-of-work-buys and annual-earnings.
- I initially added duplicate weekly-hours facts and a new source, not realizing both rooms already had them. The gate caught the duplicates immediately, which is a testament to its value — but it cost a round-trip. I should have grep'd for existing facts before adding new ones.

## Gaps to flag

- **The 1950s weekly-earnings displayed value changed from $53.29 to $53.46.** This is the honest computation from the stored inputs, but it differs from the BLS's own weekly-earnings series ($53.33). The notes explain the discrepancy. A future session could transcribe the BLS series value directly (as a regular fact, not a derived fact) if precision matters more than the derivation principle.
- **`all_facts` in `check.py:_check_derived` is rebuilt per room.** At 13 rooms and 453 facts this is negligible, but if the corpus grows large, hoisting it outside the room loop is a minor optimization.
- **The `evaluate_room` double-compute in `build.py`** (once for the manifest, once inside `project_room`) is a pre-existing redundancy. WI-5 didn't introduce it, but the new `fact_index` is built in both places. A future refactor could compute once and pass the results through.
- **The `us-1950s-purchasing-power` derived fact computes 13.02, but its `unit` field still says "CPI-U ratio (313.7 / 24.1)".** The unit is prose, not gate-checked, and the computed value is correct. But a visitor reading "313.7 / 24.1" would expect 13.03, not 13.02. The discrepancy is floating-point: 313.7 / 24.1 = 13.0249... which rounds to 13.02. The old authored "13.03" was a hand-rounded approximation. Not wrong, just worth knowing.
- **Derived facts cannot reference other derived facts.** The `fact_index` only includes `room.facts`, not `room.derived`. This is intentional (prevents transitive dependency chains), but it means a future derived fact that wants to compute, say, real weekly earnings (weekly-earnings × CPI ratio) would need both inputs as regular facts.
